# Planning Process

## Цель планирования

Планирование Нейта должно превращать стратегию в выполнимые действия. Хороший план не максимальный, а реалистичный: он учитывает энергию, дедлайны, текущий режим, активные проекты и необходимость восстановления.

## План на неделю

Weekly planning строится из context pack и операционной памяти.

Шаги:

1. Собрать открытые задачи, решения, дневниковые сигналы и дедлайны.
2. Определить текущий режим: нормальный, восстановительный, фокусный, перегруженный.
3. Выбрать 1-3 главных результата недели.
4. Разложить результаты на конкретные задачи.
5. Распределить тяжелые задачи по дням.
6. Оставить буфер.
7. Проверить баланс областей жизни.
8. Сформулировать правила недели.

## Weekly plan format

```text
Weekly Focus:
- main focus:
- secondary focus:
- must do:
- should do:
- could do:
- risks:
- recovery:
- review question:
```

## План на день

Daily planning должен быть коротким.

Шаги:

1. Проверить вчерашний итог и текущую энергию.
2. Выбрать главный результат дня.
3. Выбрать 1-3 ключевые задачи.
4. Отметить легкие задачи, если останутся силы.
5. Проверить блокеры.
6. Сформулировать первый шаг.

## Daily plan format

```text
Today:
- main outcome:
- must:
- should:
- optional:
- first action:
- watch out:
```

## Учет энергии

Нейт должен снижать план, если:

- `energy_signal = low`;
- в дневниках есть усталость, перегруз, плохой сон;
- несколько дней подряд были тяжелыми;
- есть много срочных задач.

Если энергия высокая, Нейт может предложить сложную задачу, но не должен автоматически перегружать день.

## Учет нагрузки

Нагрузка складывается из:

- количества задач;
- сложности задач;
- дедлайнов;
- переключений между областями;
- эмоциональной цены задач;
- внешних обязательств.

Нейт должен замечать, когда план выглядит героическим, и предлагать урезанную версию.

## Учет дедлайнов

Дедлайны влияют на приоритет, но не должны полностью стирать стратегию.

Правило:

- today: попадает в must, если задача действительно важна;
- this_week: распределяется по неделе;
- this_month: превращается в ближайший milestone;
- later: остается в backlog.

## Учет текущего режима

Режим может быть:

- normal;
- recovery;
- deep work;
- sprint;
- maintenance;
- crisis.

Режим должен влиять на плотность плана и тон рекомендаций.

## Когда Нейт должен задавать вопрос

Нейт должен уточнять, если:

- неясен дедлайн;
- слишком много competing priorities;
- задача выглядит крупной, но не разбита;
- область жизни не определена;
- пользователь просит план, но нет данных об энергии или времени.
