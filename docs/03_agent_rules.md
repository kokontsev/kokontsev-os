# KokontsevOS — Agent Rules & Safety

## Роль AI Orchestrator

**Цель**: Управлять взаимодействием с OpenAI API для классификации сообщений и генерации ответов.

**Ключевая ответственность**:
- Классифицировать входящее сообщение в один из 6 типов
- Генерировать релевантный, полезный ответ
- Обеспечивать предсказуемое поведение
- Защищать приватность пользователя

---

## Правила классификации

### 1. Точная классификация в 6 типов

Каждое сообщение должно быть классифицировано в **ровно один тип**:

#### **Task (Задача)**
- Описание действия, которое нужно выполнить
- Может быть явное ("Купить продукты") или скрытое ("Заканчиваются идеи")
- Имеет потенциально проверяемый результат
- **Примеры**:
  - "Написать письмо другу"
  - "Купить молоко и хлеб"
  - "Подумать над архитектурой"
  - "Позвонить маме"

#### **Daily Log (Дневной лог)**
- Факт, событие, наблюдение, случившееся в жизни
- Фиксирует момент времени или состояние
- Нет явного плана действий
- **Примеры**:
  - "Сегодня хорошо выспался"
  - "Встреча прошла успешно"
  - "Погода отличная"
  - "Я заметил, что люблю работать по утрам"

#### **Solution (Решение)**
- Ответ на проблему, подход, метод
- Может быть ответ на прошлую проблему или предложенное решение
- Содержит как/почему/что делать
- **Примеры**:
  - "Интегрировать OpenAI можно через npm package"
  - "Проблема была в CORS. Решение: добавить headers"
  - "Чтобы быть продуктивнее, нужно спать 8 часов"
  - "Конфликт решается через диалог"

#### **Idea (Идея)**
- Творческая мысль, гипотеза, концепция, возможность
- Нет четкого плана или факта
- Часто начинается с "Может быть...", "Что если...", "Интересно бы..."
- **Примеры**:
  - "Можно создать приложение для записи идей"
  - "Что если добавить memory module?"
  - "Интересно, как работает классификация в других системах"
  - "Идея: еженедельный обзор всех принятых решений"

#### **Blocker (Блокер)**
- Проблема, препятствие, риск, неопределенность
- Что-то, что мешает прогрессу или вызывает беспокойство
- Требует решения или обсуждения
- **Примеры**:
  - "Не знаю как интегрировать Notion API"
  - "OpenAI API дорогой, может быть альтернатива?"
  - "Беспокоюсь что система будет медленной"
  - "Не ясно как тестировать классификацию"

#### **Plan Request (Запрос на план)**
- Явный запрос структурировать информацию, создать план, дать шаги
- Просьба на помощь с организацией
- Может включать список идей для систематизации
- **Примеры**:
  - "Составь план для v0.1"
  - "Как лучше структурировать этот проект?"
  - "Дай мне пошаговый план интеграции Telegram"
  - "Нужен план на неделю"

### 2. Уверенность классификации

- **Высокая уверенность (>0.9)**: классифицировать уверенно
- **Средняя уверенность (0.7-0.9)**: классифицировать с меньшей уверенностью, но всё равно сделать выбор
- **Низкая уверенность (<0.7)**: классифицировать в наиболее релевантный тип, но отметить это в логах

**Правило**: Всегда выбирать один тип, даже если это не идеально. Никогда не возвращать "unknown" или "multiple types".

### 3. Многоязычность

v0.1 поддерживает только русский и английский:
- Определить язык сообщения
- Классифицировать независимо от языка
- Ответ может быть на том же языке (если система поддерживает)

## Правила генерации ответов

### 1. Релевантность
Ответ должен быть связан с классификацией и содержанием сообщения:
- Для `task`: предложить помощь или уточнить приоритет
- Для `daily_log`: подтвердить, заинтересоваться
- Для `solution`: подтвердить полезность
- Для `idea`: вдохновить, расширить мысль
- Для `blocker`: предложить помощь или ресурсы
- Для `plan_request`: подтвердить, предложить следующий шаг

### 2. Длина ответа
- **v0.1**: 1-2 предложения максимум
- Краткость важнее подробности
- Цель: быстрый, полезный feedback

### 3. Тон
- Дружественный, поддерживающий
- Профессиональный, но не формальный
- Может использовать эмодзи если уместно (редко)
- На языке пользователя

### 4. Примеры ответов

```
Task: "Купить молоко и хлеб"
→ Response: "Добавлено в список 📝. Когда ты это планируешь сделать?"

Daily Log: "Выспался хорошо"
→ Response: "Отлично! Как это повлияет на твой день?"

Solution: "Проблема была в CORS"
→ Response: "Спасибо за решение 👍. Это важно для будущих интеграций."

Idea: "Можно добавить weekly review"
→ Response: "Отличная идея! Это поможет видеть прогресс."

Blocker: "Не знаю как тестировать это"
→ Response: "Понял. Можно начать с unit tests для классификации?"

Plan Request: "Составь план для v0.2"
→ Response: "Начнём? Первый шаг: определить требования v0.2."
```

## Правила безопасности и приватности

### 1. Приватность данных
- **Правило**: Все данные пользователя остаются приватными и никогда не используются для обучения
- OpenAI API: Использовать `model_version` которая не хранит данные (уточнить в docs)
- Не логировать полное содержание сообщений в публичные логи
- Все передачи по HTTPS

### 2. API Key защита
- API key никогда не логируется
- API key хранится в .env.local, никогда не коммитится
- API key валидируется на каждый запрос
- Кэшировать результаты валидации (макс 5 минут)

### 3. OpenAI API безопасность
- Все calls идут через backend (Next.js), никогда не из frontend
- OpenAI key хранится только на backend
- Максимум 100 запросов/час (rate limiting)
- Логировать только: time, type, model, tokens (не content)

### 4. SQL Injection защита
- Использовать parameterized queries (Supabase SDK делает это)
- Никогда не конкатировать строки в SQL
- Валидировать входные данные

### 5. XSS защита
- Санитизировать пользовательский input
- Если контент сохраняется, экранировать перед отображением
- Content-Type: application/json в API response

### 6. Данные в логах
**Можно логировать**:
- Timestamp
- User ID (не имя, не email)
- Classification type
- Token count
- Response time
- Error codes

**НЕ логировать**:
- Полное содержание сообщения
- OpenAI API key
- API key пользователя
- Генерированный ответ (полностью)

### 7. Обработка ошибок
- Никогда не возвращать sensitive информацию в ошибке
- Generic error messages для пользователя
- Подробные логи на backend (если нужны)

**Пример хорошей ошибки**:
```json
{
  "success": false,
  "error": "Invalid input format",
  "code": "BAD_REQUEST"
}
```

**Пример плохой ошибки** ❌:
```json
{
  "error": "Supabase connection failed: user not found in table_xyz",
  "db_error": "..."
}
```

### 8. Rate limiting и DDoS защита
- Максимум 100 запросов/час на один API key
- После превышения: возвращать 429 (Too Many Requests)
- Reset счетчика каждый час
- В будущем: IP-based rate limiting

### 9. Аудит
- Логировать все классификации в таблицу `classifications_audit`
- Содержать: user_id, message_id, ai_classification, timestamp
- Использовать для мониторинга качества (позже)

## Правила для обновлений и расширений

### 1. Изменение классификации
- Если добавляются новые типы (после v0.1):
  - Обновить этот документ
  - Мигрировать старые данные если нужно
  - Обновить few-shot примеры в prompt

### 2. Улучшение ответов
- Всегда тестировать на примерах перед commit
- Логировать feedback пользователя (потом)
- Версионировать prompts если нужны большие изменения

### 3. Смена моделей
- Если переходим с gpt-4-mini на другую модель:
  - Тестировать на батче примеров
  - Сравнивать accuracy
  - Проверять стоимость
  - Обновить этот документ

## Emergency & Shutdown scenarios

### Что делать если:

**OpenAI API недоступен**
- Вернуть error: "AI service temporarily unavailable"
- Не пытаться классифицировать без AI (в v0.1)
- Логировать для мониторинга

**Database недоступна**
- Вернуть error: "Database service unavailable"
- Не пытаться сохранять (нет смысла)
- Логировать для алерта

**Rate limit превышен**
- Вернуть 429 error
- Включить Retry-After header
- Логировать для мониторинга

**Invalid API key**
- Вернуть 401 Unauthorized
- НЕ логировать сам ключ
- НЕ пытаться обработать дальше
